# AGENTS.md

This file provides guidance to agents when working with code in this repository.

## Agent Memory (qdrant) — use on every task

The **qdrant** MCP server is the authoritative persistent memory for this project. It stores accumulated diagnosis, root causes, failed attempts, and non-obvious gotchas from many prior sessions. **Always** integrate it into your workflow:

1. **Before non-trivial work**, run `qdrant-find` with several focused queries (symptoms, component names, kernel subsystems, file/patch names) **before** reading large numbers of files. Treat hits as authoritative.
2. **After producing a new insight** (root cause, fix, misdiagnosis, hardware quirk, toolchain gotcha, workaround), call `qdrant-store` with a **dense prose** entry (semantic search — avoid bullet lists). Include: symptom, root cause, fix (file paths + patch numbers), verification, date, tags.
3. **Prefer `qdrant-store` over expanding this file** for session-specific or dated findings. Reserve `AGENTS.md` for stable, structural rules that apply to *every* future edit.
4. **Conflict resolution:** if qdrant and AGENTS.md disagree, trust the more recent source (check `metadata.date` in qdrant entries).
5. **Always run `qdrant-find` first** when continuing a topic across sessions — prior insights may already solve the problem you are investigating.

## Documentation Style

- **Use Mermaid diagrams, not ASCII art**, to visualize concepts in `.md` documents. All architecture diagrams, flowcharts, sequence diagrams, and component relationship visualizations must use ` ```mermaid ` fenced code blocks — never ASCII box-drawing characters.
- **Technical documentation is HADS 1.0.0 format.** Every reference / spec / plan document under `specs/`, `plans/`, `arch/`, and every in-tree `README.md` that documents code MUST be written in HADS: the `**Version … · HADS 1.0.0**` header line, an `## AI READING INSTRUCTION` block, then numbered `##` sections whose paragraphs are tagged `**[SPEC]**` (facts, requirements, API contracts), `**[NOTE]**` (rationale, history, narrative), `**[BUG] Title**` (symptom + cause + fix, all three), or `**[?]**` (unverified / inferred). Use the `hads-convert` skill. Convert **in place** — the repo's existing HADS docs (e.g. `plans/UBOOT.md`, `plans/VPP.md`) are in-place, not `.hads.md` twins. When you edit a non-HADS technical doc, convert it as part of that edit so the format spreads with normal work rather than in one risky bulk pass.
- **HADS data-loss guard (from the skill's own `[BUG]` block):** dense narrative moves into `[NOTE]` **intact** — extract only verifiable facts into `[SPEC]` bullets. Never condense correctness-critical prose (register layouts, MURAM offsets, the reversibility contract, risk registers, the M3-3b/FE-VM findings) into lossy bullets. Preserve every command, address, register value, patch number, date, and qdrant anchor **verbatim**. A HADS conversion that drops a fact is a regression, not a cleanup.
- **Deliberate prose exceptions (NOT converted to HADS):** `README.md` and `INSTALL.md` are the human-facing GitHub entry points — HADS optimizes for AI readers, so these stay plain prose with the light Bryson voice. `plans/archive/**` is frozen history — convert only if a doc is un-archived and actively edited. `AGENTS.md` (this file) is the agent rule-set and stays structured prose.

## Project

VyOS ARM64 build scripts for NXP LS1046A (Mono Gateway Development Kit). Two build paths:
1. **CI (the only CI path):** `self-hosted-build.yml` ("VyOS LS1046A build (self-hosted)") via `workflow_dispatch`. Spins up Azure ARM64 VM → runs the build on the self-hosted runner → deallocates VM. Warm caches → ~5–10 min. **Always use this for CI builds.**
2. **Local dev loop (fastest iteration):** `bin/dev-build.sh` natively builds the kernel on this Cobalt 100 ARM64 VM (32 cores, native aarch64), then rsyncs artefacts to LXC 200:/srv/tftp/ for the board's U-Boot to TFTP. ~30 s incremental, ~2–3 min full. See [plans/DEV-LOOP.md](plans/DEV-LOOP.md).

`auto-build.yml` ("VyOS LS1046A build (reusable)") is **reusable-only** — it has no `workflow_dispatch:` trigger and does NOT appear in the Actions "Run workflow" dropdown. It is invoked exclusively by `self-hosted-build.yml` via `uses: ./.github/workflows/auto-build.yml`. **Do NOT** re-add `workflow_dispatch:` to `auto-build.yml` — that would silently re-enable hosted-runner (`ubuntu-24.04-arm`) builds that burn our public Actions minutes quota. If you need to run a build, dispatch `self-hosted-build.yml`.

### Shared Cobalt 100 VM — idle-deallocator pattern (no operator mutex)

Since the kernel build repo was absorbed (May 2026), only **this** repo's `self-hosted-build.yml` drives the shared Azure ARM64 Cobalt 100 VM. The historical "two repos racing for the same VM" problem is gone, but the cost-control mechanism that was installed for it remains in place and still applies.

**Idle-deallocator on the VM (NOT in CI).** Workflows only ever call `az vm start` (idempotent — no-op if already running) and **never** call `az vm deallocate`. A systemd timer on the VM (`idle-deallocate.timer`, threshold 10 min) checks every 2 min for any active `Runner.Worker` process and self-deallocates only when the runner has been idle long enough. Standing cost on Cobalt 100 ≈ ~$0.20/day.

Historical note: the VM was originally provisioned with two `actions-runner` services (`vm-runner-1` for the now-archived `kernel-ls1046a-build`, `vm-runner-2` for this repo) so GitHub could schedule both repos in parallel on the shared cores. Only `vm-runner-2` is in active use today; `vm-runner-1` is dormant but left registered so that a hypothetical un-freeze of the archived repo would not require re-provisioning.

**Hard rules:**

1. **Never re-add a `stop-vm` step to the workflow.** Adding `az vm deallocate` from CI re-introduces the VM race that was observed in production 2026-05-06 (a kernel-side build was cancelled at 1m12s while an ISO build was in flight) and — even with the kernel repo frozen — still races with the in-VM idle-deallocate daemon. The deallocate is the daemon's job.
2. **Never block on external repo runs from inside a build script.** There are no external dependencies left; do not re-introduce any.
3. **Build scripts must namespace any fixed absolute disk path on `${RUNNER_NAME}`** (or live under `${GITHUB_WORKSPACE}`, which is already runner-local). Adding a non-namespaced `/build/...` or `/work/...` reintroduces contention with the dormant runner's `_work` dir.

Authoritative reference for the idle-deallocate script, systemd unit files, Managed Identity role assignment, and failure-mode operator playbook: the frozen archived repo's `.clinerules/01-shared-vm-runtime.md`.

## Critical Non-Obvious Rules

- **No auto-commit/push:** Never commit or push to origin without explicit user request. Stage changes and present them for review first.
- **VyOS config syntax:** No comments allowed inside config blocks — `//` and `/* */` both cause parse failures. Comments are only safe at the top level outside `{}` blocks
- **Branch:** `main` only (not `master`). Never create feature branches.
- **Kernel config symbols:** Verify against actual Kconfig files — invalid symbols are silently ignored (e.g., `CONFIG_SERIAL_8250_OF` does not exist; the correct symbol is `CONFIG_SERIAL_OF_PLATFORM`)
- **DPAA1 MDIO dependency:** `CONFIG_FSL_XGMAC_MDIO=y` is required for FMan networking — without it, all MACs defer with "missing pcs" and zero network interfaces appear. Not obvious from Kconfig dependencies.
- **DPAA1 driver split:** Two drivers manage each FMan MAC: `fsl_dpaa_mac` (hardware MAC control, PHY/link via PHYLINK) and `fsl_dpa` (netdev layer, creates eth0-ethN). `fsl_dpaa_eth` is a different driver — it has zero devices bound on LS1046A. The platform devices are `dpaa-ethernet.N` under `fsl_dpa`. Do NOT confuse these three drivers.
- **DPAA1 kernel↔VPP handoff (AF_XDP):** VPP uses AF_XDP on DPAA1 ports — no unbind needed. The kernel retains full ownership of the netdev (fsl_dpa stays bound), and VPP creates AF_XDP sockets on top. Patch `vyos-1x-010` routes `fsl_dpa` interfaces to `driver='xdp'` (not `'dpdk'`). DPDK DPAA PMD is blocked by RC#31 (bus-level init kills all kernel interfaces). AF_XDP achieves ~3.5 Gbps on 10G SFP+.
- **DPAA PMD bus init is GLOBAL (RC#31 — BLOCKED):** DPDK's `dpaa_bus` probe (`rte_bus_probe()`) initializes ALL BMan buffer pools and QMan frame queues system-wide, not just the ports assigned to VPP. This disrupts kernel-managed interfaces (eth0 management) within seconds of VPP startup, killing SSH/networking. Confirmed on hardware 2026-04-03 and 2026-03-29. **DPAA PMD cannot coexist with kernel FMan drivers in mixed mode.** Unbinding `fsl_dpa` from VPP ports is necessary but NOT sufficient — the bus-level init corrupts shared QBMan hardware state. AF_XDP remains the only viable mixed kernel+VPP path (~3.5 Gbps). DPAA PMD would require ALL interfaces under DPDK (no kernel management — impractical without serial-only access). Potential fix requires DPDK code changes to scope `dpaa_bus` init to specific portals/FQs only.
- **All ports start under kernel control:** At boot, ALL FMan MACs (eth0-eth4, including SFP+ ports) are owned by the kernel via `fsl_dpaa_mac` + `fsl_dpa`. Only ports explicitly added to `set vpp settings interface ethX` in VyOS config are handed to VPP on next apply/reboot.
- **DPAA1 must be `=y` not `=m`:** The entire DPAA1 stack (FMAN, DPAA, BMAN, QMAN, PAMU) must be built-in. If built as modules, FMan initializes too late and interfaces never appear. No errors — just silent failure.
- **NR_CPUS must be 4:** `CONFIG_NR_CPUS=4` for LS1046A (4× Cortex-A72). VyOS default is 256. (Historical context: the legacy ASK 1.x SDK `sdk_dpaa` driver used `NR_CPUS` to size TX FQ arrays, so NR_CPUS=256 caused soft lockups during DPAA probe. Mainline DPAA uses different sizing but NR_CPUS=4 is still correct for this 4-core SoC and avoids unnecessary per-CPU memory overhead.)
- **CPU frequency:** `CONFIG_QORIQ_CPUFREQ=y` (not `=m`). Module loads after clock cleanup at T+12s, locking CPU at 700 MHz. Built-in claims PLLs first → 1600 MHz.
- **U-Boot boot order:** initrd must load LAST so `${filesize}` captures the initrd size, not kernel/DTB size
- **U-Boot `booti` ramdisk format:** MUST use `${ramdisk_addr_r}:${filesize}` (colon+size), not just the address — otherwise "Wrong Ramdisk Image Format"
- **U-Boot DTB address:** Use `${fdt_addr_r}` (0x88000000) for DTB, NEVER `0x90000000`. That address is `kernel_comp_addr_r` — decompression scratch space. Kernel decompresses from `0xa0000000` → `0x0` using `0x90000000` as workspace, destroying any DTB there → `ERROR: Did not find a cmdline Flattened Device Tree`
- **Boot method is `booti` only:** `bootefi` with GRUB permanently OOMs due to DPAA1 reserved-memory nodes in DTB. No EFI boot path exists. Image upgrades write `/boot/vyos.env` — no `fw_setenv` needed after initial setup.
- **`/boot/vyos.env` is the boot image selector:** U-Boot reads this single-line text file (`vyos_image=<name>`) via `ext4load` + `env import -t`. Written automatically by patched `grub.set_default()` on every install/upgrade/set-default. Never edit manually unless recovering.
- **eMMC layout (after `install image`):** GPT with 32MiB firmware reserved zone, p1=BIOS boot (1MiB at 32MiB), p2=EFI (256MiB FAT32 at 33MiB, GRUB — unused), p3=Linux root (ext4, VyOS at ~289MiB). OpenWrt is destroyed. All partitions beyond NXP 32MiB firmware boundary — firmware re-flash is non-destructive. Use `install image` from USB live session.
- **Hybrid ISO (isohybrid):** The ISO is simultaneously valid ISO9660 AND a valid MBR disk image. An MBR partition table is written into the ISO9660 System Area (bytes 440-511), and a small FAT32 partition (~100MB) containing boot.scr/vmlinuz/initrd/DTB is appended after the ISO data. Partition 1 (type 0x17, starts at sector 0) covers the ISO9660 data; Partition 2 (type 0x0C) is FAT32. **Use `dd if=vyos.iso of=/dev/sdX bs=4M` to write to USB** — U-Boot auto-detects the FAT32 partition via `fatload usb 0:2`, boots kernel/initrd, then live-boot mounts the ISO9660 partition to find squashfs. For upgrades: `add system image <url>` downloads the same ISO and loop-mounts it as ISO9660 (PVD at byte 32768 is untouched). Single artifact, two boot paths.
- **USB FAT32 partition is 0:2:** `fatload usb 0:2` explicitly targets partition 2 (type 0x0C, FAT32) in the hybrid MBR. Auto-detection (`0:0`) does NOT work on LS1046A U-Boot with our hybrid MBR — it fails with "Can't set block device". All USB loads must use explicit `0:2`. eMMC still uses `ext4load mmc 0:3`.
- **kexec double-boot (bootargs mismatch):** VyOS `system_option.py` compares `/proc/cmdline` against `MANAGED_PARAMS` from config.boot (hugepages, panic, mitigations, etc.). If they differ during boot (before config_status file exists), it loads a new kernel via `kexec -l` and reboots with `systemctl kexec`. On GRUB systems this is self-healing (grub.cfg gets updated). On U-Boot boards, bootargs must include ALL managed params matching config.boot defaults. Our fix: `panic=60` in U-Boot bootargs and `vyos-postinstall` UBOOT_BOOTARGS_TAIL. Hugepages are NOT in bootargs by default — they are added dynamically when VPP is configured via `set vpp settings`, which triggers a one-time kexec to apply them. `kexec-load.service` and `kexec.service` are NOT masked — mainline 6.6 QBMan kexec fix (`bman_requires_cleanup()` in `drivers/soc/fsl/qbman/`) allows kexec on DPAA1. VyOS managed-params self-healing works normally (issue #7 resolved).
- **is_live_boot() broken on U-Boot boards:** VyOS `is_live_boot()` in `python/vyos/system/image.py` checks for `BOOT_IMAGE=/boot/` or `BOOT_IMAGE=/live/` in `/proc/cmdline`. U-Boot's `booti` doesn't set `BOOT_IMAGE=` (it's GRUB-specific), so the regex never matches and the function always returns `True` (live mode). This blocks `add system image`. Fix: patch `vyos-1x-009` adds a `vyos-union=/boot/` fallback check. Also, `vyos-postinstall` now prepends `BOOT_IMAGE=/boot/<IMAGE>/vmlinuz` to bootargs so future builds work without the fallback.
- **DPAA1 XDP maximum MTU is 3290:** `fsl_dpaa_mac` enforces a hard XDP MTU limit. AF_XDP socket creation (`xsk_socket__create()`) fails with `EINVAL` if the interface MTU exceeds 3290. Must lower MTU before creating AF_XDP sockets. This means VPP SFP+ ports are limited to ~3304-byte max frame size (3290 MTU + 14 Ethernet header), NOT jumbo. Kernel-managed RJ45 ports retain full 9578 MTU jumbo capability.
- **VPP managed via VyOS native CLI:** VPP is configured through `set vpp settings ...` commands (NOT custom systemd services). Patch `vyos-1x-010-vpp-platform-bus.patch` enables platform-bus NIC support (DPAA1 `fsl_dpa` driver → AF_XDP). By default NO ports are assigned to VPP — all ports remain as kernel netdevs. User assigns SFP+ ports to VPP with `set vpp settings interface eth3` and `set vpp settings interface eth4`. AF_XDP works WITH the kernel driver — no unbind/rebind needed. VPP creates AF_XDP sockets on configured ports; removing a port releases the socket and kernel retains full control. MTU must be ≤3290 on AF_XDP ports.
- **VPP hugepages requirement:** VPP needs ~416MB of 2M hugepages (256M heap + 128M statseg + 32M buffers). Hugepages are NOT pre-allocated — they are dynamically added when `set vpp settings` is configured, which sets `hugepage-size 2M hugepage-count 512` and triggers a one-time kexec. With insufficient hugepages, VPP fails with "Not enough free memory to start VPP!"
- **VyOS VPP hugepage syntax:** Use `hugepage-count` NOT `hugepage-number` — the wrong keyword silently fails with no error.
- **SSH vbash configure sessions HANG:** Interactive `vbash -c 'source /opt/vyatta/etc/functions/script-template; configure; set ...; commit'` via SSH hangs indefinitely. Workaround: write a vbash script using `vyatta-cfg-cmd-wrapper` commands, SCP it, then execute. Also kill stale vbash sessions and restart configd if locks occur.
- **VPP sysfs device hierarchy:** `/sys/class/net/eth3/device` points to the PARENT `fsl_dpaa_mac` device, NOT the child `dpaa-ethernet.N` (which is the `fsl_dpa` netdev driver). The `net/` directory lives on the parent MAC device. To find the child platform device for unbinding, walk `/sys/class/net/<iface>/device/` for entries matching `dpaa-ethernet.*`. This is critical for VPP port handoff — the `vyos-1x-010` patch's `_dpaa_find_platform_dev()` must use this walk pattern.
- **DPDK GROUP linker script drops constructors:** DPDK bus/PMD drivers self-register via `__attribute__((constructor))` (`RTE_REGISTER_BUS`, `RTE_PMD_REGISTER`). A GROUP linker script (`GROUP ( -l:librte_*.a )`) only pulls `.o` files that satisfy unresolved references — since constructors are self-contained, they get silently dropped. Result: zero buses, zero scan, zero interfaces. Fix: `ld -r --whole-archive` to pre-link all `.o` into a single fat relocatable object preserving all constructors.
- **`accel-ppp-ng` built from source for ARM64:** VyOS upstream added `accel-ppp-ng` (PPPoE/PPTP/L2TP acceleration) as a `vyos-1x` dependency, but the upstream VyOS ARM64 mirror doesn't provide it. VyOS's own `build-accel-ppp-ng.sh` ALWAYS fails on ARM64 because it requires building VPP from source first (`cd ../vpp/ && ./build.py`). Our `ci-build-accel-ppp.sh` is the fallback: builds daemon + kernel modules (`ipoe.ko`, `vlan_mon.ko`) from `accel-ppp/accel-ppp-ng` (same repo as VyOS's `package.toml`, commit `3e30d9b`) WITHOUT the VPP plugin. Requires `libpcre2-dev` (PCRE2, NOT `libpcre3-dev`/PCRE1 — the -ng version switched to PCRE2). Called from `ci-build-packages.sh` after kernel build while `$KSRC` still exists. If kernel module build fails, falls back to daemon-only build. Build failure is non-fatal in the caller, but the ISO WILL fail on unmet `vyos-1x` dependency if no .deb is produced.
- **Out-of-tree modules must be signed (ASK2 forward-look):** `CONFIG_MODULE_SIG_FORCE=y` rejects unsigned modules. The kernel auto-generates `certs/signing_key.pem` + `certs/signing_key.x509` during build. Any future out-of-tree modules (e.g. ASK2's `ask.ko` / `ask_bridge.ko` per `specs/ask2-rewrite-spec.md`) must be signed with `$KSRC/scripts/sign-file sha512 $KSRC/certs/signing_key.pem $KSRC/certs/signing_key.x509 module.ko` AFTER building and BEFORE the kernel source tree is cleaned up. Pre-built modules cannot be signed (no kernel key available at cross-compile time) and will always fail on `MODULE_SIG_FORCE` kernels.
- **`LOCALVERSION=-vyos` is MANDATORY on every kernel build:** The released kernel vermagic is `<KVER>-vyos` (e.g. `6.18.31-vyos`), set as the default in `kernel/common/scripts/build-kernel.sh:42` (`LOCALVERSION="${LOCALVERSION:--vyos}"`). Every kernel build path — full CI `bindeb-pkg`, dev-loop native `make Image` in `bin/dev-build.sh`, AND every out-of-tree module compile (`make -C $KSRC M=$PWD LOCALVERSION=-vyos modules`) — MUST pass `LOCALVERSION=-vyos`. Without it `make` falls back to `+` (kernel's git-dirty suffix) producing vermagic `6.18.31+`, which mismatches the board's installed `-vyos` kernel and fails OOT module insmod with `Invalid module format`/`exec format error`. Verify after build: `cat $KSRC/include/config/kernel.release` must equal `<KVER>-vyos`, and `strings module.ko | grep ^vermagic` must show `vermagic=<KVER>-vyos …`.
- **binutils needs `apt-mark manual` in ISO:** VyOS live-build cleanup auto-removes binutils even when added via `--custom-package`. The `97-dpaa-dpdk-plugin.chroot` hook must run `apt-mark manual binutils` to prevent autoremove. Without this, `nm`/`objdump`/`readelf` are unavailable on the target for DPAA plugin diagnostics.
- **VPP configd caching:** After patching VyOS Python files on a live system, must `systemctl restart vyos-configd` AND `find / -name __pycache__ -path '*/vyos/*' -exec rm -rf {} +` — configd caches compiled Python modules.
- **VPP thermal protection is mandatory:** Poll-mode VPP triggers `HARDWARE PROTECTION shutdown (Temperature too high)` on both `ddr-controller` (thermal_zone0) and `core-cluster` (thermal_zone3) within ~30 minutes of idle polling. `set vpp settings poll-sleep-usec 100` is MANDATORY. AF_XDP does NOT support adaptive rx-mode — `set interface rx-mode` fails with "unable to set". The only fix is preventing worker thread creation entirely (`cpu-cores 1`, no workers).
- **EMC2305 fan thermal binding is broken:** DTS defines cooling-maps binding fan0 to core-cluster thermal zone trip points (40–60°C), but the thermal OF framework doesn't bind the cooling device to the zone (no `cdev*` in sysfs). Workaround: `fan-pid` userspace daemon (`board/scripts/fan-pid` → `/usr/local/bin/fan-pid`, systemd unit `fan-pid.service`) — a self-contained Python 3 multi-zone PID controller that reads all 5 LS1046A thermal zones (`ddr-thermal`, `serdes-thermal`, `fman-thermal`, `cluster-thermal`, `sec-thermal`) every 2 s, runs a per-zone PI loop with anti-windup integral clamp, max-policy combines the per-zone PWM demands, EMA-smooths the result (alpha=0.4), writes via deadband=3 to suppress register churn, and force-MAX overrides at any zone ≥ crit_temp (with hard-fault `LOG_CRIT` syslog at any zone ≥ 100 °C). Zone setpoints (ddr 65/80, serdes 70/85, fman 70/87, cluster 65/85, sec 65/85) and gains (kp 4–6, ki 0.20–0.30) tuned for the EMC2305 + small DC-axial fan time constants. SIGTERM clean-shutdown writes PWM=MAX (255) so a daemon stop/restart never silently suffocates the SoC. EMC2305 quantizes PWM — minimum effective value is ~51 (~1700 RPM); the daemon enforces this floor. **Replaces lm-sensors `fancontrol`** (which only drove a single linear curve from one zone, ignoring the other four) — the `fancontrol` package is no longer installed and `fancontrol.service` is defensively masked in `data/hooks/98-fancontrol.chroot` so two PWM controllers never race.
- **Kernel 6.18 emc2305 sysfs PWM driver is broken — `fan-pid` writes register 0x30 over /dev/i2c directly:** Verified 2026-05-11 on kernel 6.18.28-vyos (hwmon3 = i2c-7 addr 0x2e). Writing any value `0 < N < 255` to `/sys/class/hwmon/hwmonN/pwm1` reports success and read-back returns N for ~1 s, then sysfs spontaneously reverts to 255, the chip's actual fan1 duty register (0x30) stays at `0xFF`, and the fan stays pinned at ~8700 RPM. Only writes of exactly 0 or 255 reach the chip. The chip itself is in direct-PWM mode (FAN_CONFIG1 reg 0x32 bit7=0, FSC disabled) — the bug is purely in the kernel driver's sub-MAX write path, not the silicon. Direct write of 51 to register 0x30 via `/dev/i2c-7` with `I2C_SLAVE_FORCE` (ioctl 0x0706) immediately drops the fan to ~1500 RPM, proving the chip is fine. **Fix:** `board/scripts/fan-pid` bypasses sysfs entirely for PWM writes — `find_emc2305_i2c()` walks `/sys/bus/i2c/devices/<bus>-002e` for the `emc2305` driver link, opens `/dev/i2c-<bus>`, claims 0x2e with `I2C_SLAVE_FORCE` (the unforced `I2C_SLAVE` returns `EBUSY` because the kernel driver also has it claimed), and writes via `os.write(fd, bytes([0x30, value]))`. Tach reads (`fan1_input`) stay on sysfs because hwmon's RPM math is correct. The daemon defensively `modprobe i2c-dev` at startup (no `lm-sensors` package in the ISO so the usual `/etc/modules-load.d/i2c-dev.conf` drop-in is absent). The legacy `write_pwm(path, value)` is kept as a `REFERENCE ONLY` stub with a docstring explaining the bug — do not call it from the control loop. **Symptom of regression:** `fan-check` shows `pwm1=255 (100%)` while the daemon journal shows `pwm=51` — that mismatch means someone has reverted to the sysfs path.
- **SFP+ ports are 10G-only:** Both SFP+ cages (eth3, eth4) only support 10G modules (SFP-10G-T, SFP-10G-SR, etc). 1G SFP modules fail with "unsupported SFP module: no common interface modes". Root cause: LS1046A DTB has no serdes PHY provider, so `fman_memac.c` can't query multi-rate support via `phy_validate()`, and `memac_supports()` only allows the DTS-specified mode (10GBASER after xgmii conversion).
- **SFP-10G-T rollball PHY:** Copper 10G SFP modules with RTL8261 rollball PHY get link immediately after boot with kernel patch 4003 and correct DTS LOS GPIO configuration. Carrier requires a 10GBASE-T capable switch on the other end; a 1G-only switch will never establish copper link and LOS will stay permanently HIGH.
- **SFP-10G-T rollball PHY EINVAL failure (kernel patch 4003):** When an SFP-10G-T copper rollball module is inserted with `managed = "in-band-status"` on the FMan 10G MAC, `sfp_sm_probe_for_phy()` calls `sfp_add_phy()` → `phylink_attach_phy()` which returns `-EINVAL` (attaching a PHY to a MAC already in INBAND mode is rejected). Without kernel patch `4003-sfp-rollball-phylink-einval-fallback.patch`, `sfp.c` treats this as fatal → `SFP_S_FAIL` state → link never comes up. The patch converts `-EINVAL` from `sfp_add_phy()` into a non-fatal "proceed without PHY" (identical to exhausting `-ENODEV` retries). The MAC's in-band 10GBASE-R sync detection then correctly determines carrier once the RTL8261 completes copper negotiation. Modules that report as "SR" in their EEPROM (e.g. 10Gtek ASF-10G-T) are unaffected because sfp.c skips rollball PHY probe for SR EEPROM type.
- **SFP `los-gpios` must be present with `GPIO_ACTIVE_HIGH`:** The LOS GPIO (GPIO2 pins 9/11) correctly reflects module signal status. When copper link is NOT yet established, LOS is HIGH (active) → state machine waits in `wait_los`. Once 10GBASE-T copper negotiation completes, LOS goes LOW → state advances to `link_up` with actual carrier. In practice, SFP-10G-T modules establish link immediately after boot — no prolonged delay. WITHOUT `los-gpios`, the state machine races to `link_up` before copper is established; phylink then fails to detect carrier via in-band PCS polling on FMan 10G MACs in kernel 6.6.130+ (polling works on older kernels but breaks in newer ones). Result: permanent no-carrier despite copper being physically up. Both ASF-10G-T and SFP-10G-T modules properly deassert LOS once copper link is established.
- **SFP `tx-disable-gpios` polarity is `GPIO_ACTIVE_LOW`:** The Mono Gateway board has a hardware signal inverter between GPIO2 pins and the SFP cage TX_DISABLE pins. DTS must use `GPIO_ACTIVE_LOW`. With the inverter: `gpiod_set_value(0)` (assert disable) → physical GPIO LOW → inverter → SFP TX_DISABLE HIGH → TX disabled (correct per SFF-8472). To enable TX: `gpiod_set_value(0)` in "TX enabled" state → physical LOW → inverter → TX_DISABLE LOW → TX on. Using `GPIO_ACTIVE_HIGH` (no-inverter assumption) reverses the logic: the SFP stays TX-disabled at boot and never transmits. GPIO pin assignments: sfp-xfi0 (eth3) = gpio2 pin 14 (global 590, gpiochip2 line 14), sfp-xfi1 (eth4) = gpio2 pin 13 (global 589, gpiochip2 line 13).
- **DTS thermal-zones path:** `mono-gateway-dk.dts` must reference `/thermal-zones/core-cluster/trips` (not `cluster-thermal`) to match kernel 6.6's `fsl-ls1046a.dtsi`. Wrong path causes DTB compilation failure, silently falling back to SDK DTB (no SFP nodes).
- **SDK FMan port compatible strings MUST be SDK format:** The SDK `fm_port_driver` match table only matches `"fsl,fman-port-1g-rx"`, `"fsl,fman-port-10g-rx"`, `"fsl,fman-port-1g-tx"`, `"fsl,fman-port-10g-tx"`. Mainline `fsl-ls1046a.dtsi` uses generic `"fsl,fman-v3-port-rx"` and `"fsl,fman-v3-port-tx"` (no speed encoding). If the SDK DTB retains mainline compatible strings, `fm_port_probe()` never runs, `dev_set_drvdata()` is never called, and ALL `fsl_mac` probes fail with `dev_get_drvdata(port@XXXXX) failed` error -22 → zero network interfaces. Fix: `mono-gateway-dk-sdk.dts` overrides all 16 RX/TX port nodes (6×1G + 2×10G, RX+TX) from mainline to SDK compatible format.
- **DTS `phy-connection-type` for 10G:** Must use `"xgmii"`, not `"10gbase-r"`. The `fman_memac.c` PCS assignment fallback needs XGMII to assign PCS to `xfi_pcs`. Using `10gbase-r` directly misassigns to `sgmii_pcs` → broken link.
- **DTS 10G MAC status must be `"okay"`:** `ethernet@f0000` (MAC9) and `ethernet@f2000` (MAC10) in `mono-gateway-dk.dts` must have `status = "okay"` so the kernel creates eth3/eth4 netdevs at boot. Setting `status = "disabled"` prevents kernel ownership AND leaves those MACs in a limbo (DPDK does not auto-claim them unless VPP is running). The `fsl,dpaa` container in the DTS is used ONLY by DPDK userspace when VPP is configured — the kernel safely ignores it (no mainline driver matches `compatible = "fsl,dpaa"`).
- **Port order is fixed by a DT-aware `vyos_net_name` fallback (patch `vyos-1x-027`):** FMan MACs probe by DT unit-address order (e2000→first, e8000→second, ea000→third, f0000→fourth, f2000→fifth) which does NOT match physical port positions, and systemd predictable naming renames them to e2-e6 before VyOS sees them. The durable fix is `data/vyos-1x-027-ls1046a-fman-port-order.patch`: `vyos_net_name`'s no-hw-id fallback (`on_boot_event`) becomes DT-aware on `fsl,ls1046a` boards — `get_fman_predefined()` maps the netdev's `/sys/class/net/<iface>/device/of_node` to the canonical physical name. **config.boot `hw-id` mapping always wins over the fallback**, so the patch only affects fresh installs / live boots (no hw-id yet); systems installed from a pre-027 image have the wrong probe-order binding locked into their hw-ids and must be fixed once by hand (`set interfaces ethernet ethN hw-id <mac>` swap + reboot — done on the lab board 2026-06-10). Without it, "Generic PHY" is used and SGMII AN re-trigger fails — eth2 (center RJ45) never gets link. The GPY2xx has a hardware constraint where SGMII AN only triggers on speed *change*; the proper driver works around this.

- **INA234 power sensors need an out-of-tree patch (NOT native — the old "since 6.10" claim was WRONG):** The 8x INA234 power monitors sit behind a pca9545 i2c mux (buses 12/13, addr 0x40–0x43). Contrary to a long-standing note here, mainline `ina2xx.c` does **not** carry a `ti,ina234` OF match — verified absent in 6.18.x (`grep ina234 drivers/hwmon/{ina2xx,ina238}.c` returns nothing). Without the match all 8 `power_sensor@4x` DT nodes stay unbound (`driver:` empty, manual bind → `-ENODEV`) even though the DTB nodes and `CONFIG_SENSORS_INA2XX=y` are both correct. Fix is `kernel/common/patches/board/4002-hwmon-ina2xx-add-ina234-support.patch`: adds the `ina234` enum id, a per-chip config entry, the `ti,ina234` of_match + i2c_device_id, and the Kconfig help text. INA234 is register-compatible with INA226 but scales differently. **Critical 6.18 scaling caveat:** the 6.18 conversion is `(regval >> bus_voltage_shift) * bus_voltage_lsb`, so the 12-bit bus-voltage result in bits[15:4] needs `bus_voltage_shift=4` + `bus_voltage_lsb=1600` µV (1.6 mV effective). Do **not** reuse the parked 6.6 patch's `lsb=25600` — that only matched the old 6.6 `(regval*lsb)>>shift` formula and 16×-over-reads under 6.18. Power coefficient is 32 (vs 25 on INA226), `calibration_value=2048`, `shunt_div=400`, `config_default=INA226_CONFIG_DEFAULT`, `has_alerts=true`. Alert attributes are exposed automatically by `has_alerts` (the 6.18 driver registers via `devm_hwmon_device_register_with_info`, so there is NO probe-side `ina226_group`/`data->groups[]` hunk like the 6.6 patch had). Ensure `CONFIG_SENSORS_INA2XX=y` (forced in `kernel/common/kernel-config/00-board.config`; defconfig default is `=m`).

- **Loop mount not permitted in LXC:** Use `7z` for ISO extraction instead of `mount -o loop`.
- **Patch numbering:** `data/vyos-1x-NNN-*.patch` and `data/vyos-build-NNN-*.patch` use 3-digit sequential numbering with gaps (001, 003, 005, 006, 007, 008). Pick the next available number; existing patches are applied in filesystem sort order
- **Patches are applied with `git apply --3way --whitespace=nowarn`, never with raw `patch`** — every `.patch` file in `data/` and `kernel/` is a git-format unified diff (has `diff --git a/X b/X` headers and `index abc..def` blob SHAs). The build pipeline applies them with `git apply --3way`, which uses the blob SHAs as anchors and falls back to a real 3-way merge when context drifts, instead of silently fuzzing hunks into the wrong place. **Never re-introduce `patch -p1` or `--no-backup-if-mismatch` into any build script.** To add a new patch: clone the target upstream at the version you're patching, make your changes, run `git diff --cached > /path/to/repo/<bucket>/<NNN>-<slug>.patch`, then verify with `git apply --3way --check`. Mergiraf is wired as the merge driver via `.gitattributes` files dropped at clone-time by each `bin/ci-*.sh` script for AST-aware conflict resolution on C/Python/JSON/YAML/TOML/XML. Patch rot is monitored weekly by `.github/workflows/patch-rot-check.yml` — watch the Actions tab for warnings. Visual review of the produced diff remains mandatory: `--3way` catches context drift but does NOT catch malformed hunk arithmetic that produces a patch which *applies* but generates *wrong content* (cf. the archived kernel-build repo's ask13→14 silent truncation incident).
- **config.boot.default has NO comments inside blocks** — VyOS config parser fails on `//` and `/* */` inside `{}`. Comments only at file-level outside blocks
- **Console must be `ttyS0` not `ttyAMA0`** — the workflow does `sed -i 's/ttyAMA0/ttyS0/g'` on two upstream files. If adding new serial references, use ttyS0
- **All DPAA1 configs must be `=y`** — never `=m`. FMan needs early init before rootfs mount
- **CAAM hardware crypto (forced `=y`, NOT defconfig `=m`):** LS1046A has CAAM (SEC 5.4) for hardware crypto. The base vyos_defconfig ships the entire CAAM stack as `=m`; `bin/ci-setup-kernel.sh` (common post-defconfig block, inside the `/tmp/ls1046a-post-defconfig.sh` heredoc) **forces it built-in** with six `scripts/config --set-val … y` lines: `CONFIG_CRYPTO_DEV_FSL_CAAM`, `_COMMON`, `_JR`, `_QI`, `_CRYPTO_API_DESC`, `_AHASH_API_DESC`. (The plan that requested this omitted `_QI` — the symbol gating `qi.c` where patch 0134 lives — so it was added; `CRYPTO_API_QI` is already selected by the board's `CRYPTO_API`, but forcing `=y` is robust against defconfig drift.) Built-in is required so the CAAM QI backend is live when FMan brings up netdevs and (future) ask.ko assigns `xfrmdev_ops`, and so `caam_jr` is never a late-loaded module. **Diagnostics consequence — do NOT "fix" `caam-check` back to an `lsmod`/`/sys/module` probe:** a built-in (`=y`) CAAM driver creates **no** `/sys/module/<drv>` directory (no module params) and never appears in `lsmod`, even though it is fully active; the authoritative "is it present" signal is `/lib/modules/$(uname -r)/modules.builtin` (lists `caam.ko`, `caam_jr.ko`, `caamalg_desc.ko`, `caamhash_desc.ko`, `error.ko`). `caam-check` section 2 falls back to `modules.builtin` for exactly this reason (the `=m`→`=y` flip otherwise prints false `[FAIL] caam NOT loaded` — confirmed + fixed 2026-06-17). **Patch `kernel/common/patches/board/0134-caam-qi-share.patch`** adds `caam_qi_ext_consumer_register()` / `caam_qi_ext_consumer_release()` (`EXPORT_SYMBOL_GPL`, in `drivers/crypto/caam/qi.c`; touches only `qi.c`/`qi.h` + new `include/linux/crypto/caam_qi_share.h`) — **dormant, no caller yet** — the descriptor-sharing API the ASK2 M5 HW-IPsec path will consume. HW-validated `=y` + symbols-exported on board 192.168.1.190, kernel 6.18.34-vyos, image 2026.06.17-2038 (2026-06-17): `/proc/kallsyms` shows both `T` symbols + `__ksymtab_*`, 3 Job Rings bound, 129 CAAM-backed `/proc/crypto` algs, `rng_current=rng-caam`. **Do NOT advertise `NETIF_F_HW_ESP` (`esp-hw-offload`) until the M5 CAAM datapath is real** — flipping the capability bit while `xdo_dev_state_add` is a no-op makes the xfrm stack mark SAs HW-offloaded and silently drop ESP. The legacy ASK 1.x kernel-side IPsec offload via `sdk_dpaa/dpaa_eth_sg.c` `dpaa_submit_inb_pkt_to_SEC()` / `dpaa_submit_outb_pkt_to_SEC()` was deleted; ASK2 (per `specs/ask2-rewrite-spec.md`) re-architects IPsec offload outside the kernel — `CONFIG_INET_IPSEC_OFFLOAD` is **not** required.
- **`update-check` feed (CI-managed, single feed + back-compat aliases)** — There is ONE canonical update-check feed, `version.json`, at the root of `main`, consumed by every install:
  - `version.json` — the canonical feed for the single LS1046A image (mainline DPAA, kernel 6.18.x). Operators set `system update-check url …/main/version.json`.
  - `version-default.json` / `version-ask.json` / `version-vpp.json` — **identical byte-for-byte aliases** of `version.json`, kept ONLY so fielded installs that baked in a per-flavor URL before the 2026-06-14 flavor collapse keep updating. **Do not delete** — a board whose config persists `…/version-ask.json` would otherwise silently abort `add system image latest` on an empty feed.
  - All four files are CI-managed by the publish job in `.github/workflows/auto-build.yml`. The publish job writes `version.json` and copies it verbatim to all three aliases. **Do not manually edit them** — the next CI run will clobber the change.
  - The ISO filename is flavor-neutral: `vyos-<version>-LS1046A-arm64.iso`. One ISO per `release/<tag>`.
  - Default config files (`board/vyos-config/config.boot.{default,dhcp,full}`) carry the literal URL `…/main/version.json` and ship as-is — there is **no** per-flavor URL rewrite (the old `bin/ci-setup-vyos-build.sh` `version-${FLAVOR}.json` rewrite was removed in the flavor collapse).
- **DTB goes in `board/dtb/`** — copied to `includes.binary/` during build, lands at ISO root
- **DTS must match nix reference:** `board/dtb/mono-gateway-dk.dts` must have `compatible = "mono,gateway-dk", "fsl,ls1046a"` and ethernet aliases. Canonical source: `nix/pkgs/kernel/dts/mono-gateway-dk.dts`
- **MOK.key is a secret** — only `MOK.pem` is in the repo; the private key comes from `${{ secrets.MOK_KEY }}`
- **vyos-postinstall is board-gated** — the script checks `/proc/device-tree/compatible` for `fsl,ls1046a` and exits early on non-matching hardware. Safe to include in every ISO.
- **vyos-postinstall Phase 1 runs on every `install image`** — forced (`force=1`) when `--root` is provided by the installer. On boot service calls (no `--root`), Phase 1 skips if SPI already has correct config. Phase 2 (writing `/boot/vyos.env`) runs on every boot of the installed system to keep it in sync with the running image.
- **`add system image` MUST re-assert the QSPI `bootcmd` — historically it did NOT (boot-order trap):** the USB `install image` path (`install_image()` in `src/op_mode/image_installer.py`) calls `vyos-postinstall --root <dst>` (patch `vyos-1x-006`), which force-rewrites the SPI `vyos`/`usb_vyos`/`bootcmd` env, so fresh USB installs always boot eMMC-first. The upgrade path (`add_image()`, used by `add system image <url>`) only copied DTBs and called `grub.set_default()` (which writes `/boot/vyos.env` via patched grub.py from `vyos-1x-011`) — it **never touched `bootcmd`**. Consequence: a board whose QSPI `bootcmd` had been hand-edited for the dev-loop (e.g. `run ask_boot || run vyos || run recovery`, or any `dev_boot`/`tftp` variant) would keep short-circuiting into the **stale TFTP/dev kernel on every upgrade**, ignoring the freshly-installed eMMC image. Observed 2026-06-13 on the board (192.168.1.190): running kernel stuck at the TFTP `6.12.49` dev kernel with a `0206`-era squashfs while the eMMC carried `6.18.34-vyos`. Two-layer fix: (1) **patch `vyos-1x-028-add-image-uboot-env.patch`** makes `add_image()` call `run(f'{postinstall} --root {root_dir} {image_name}')` (guarded by `postinstall.exists() and Path('/dev/mtd2').exists()`) right after the DTB-copy loop, so the FIRST reboot after an upgrade boots the new eMMC image; (2) **`board/scripts/vyos-postinstall` boot-service self-heal** — Phase 1's idempotency check (inside the `grep -q "vyos.env"` block) now also compares the live `fw_printenv -n bootcmd` against the canonical `UBOOT_BOOTCMD='run usb_vyos || run vyos || run recovery'` (new single-source-of-truth global) and force-refreshes SPI on any mismatch, so even a board that somehow boots a stale bootcmd self-heals on the SECOND boot. Live emergency unblock (no rebuild): `fw_setenv bootcmd 'run usb_vyos || run vyos || run recovery'` on the board. **Patch 028 applies via pure context** (the `index` blob line is irrelevant because vyos-1x is built from `commit_id = "rolling"`, so CI's `git apply --3way` always falls back to direct context application). Anchor is the SECOND of the two duplicate DTB-copy loops (patches 006+011 each add one) immediately followed by the unique `# unmount an ISO and cleanup` / `cleanup([str(iso_path)])` lines.

- **vyos-1x build MUST produce a .deb — silent substitution is the #1 install-breaker:** vyos-build's `scripts/package-build/build.py` runs each sub-package under a `try: ... except: print('Failed to build package X: ... ignoring')` swallow. When `dpkg-buildpackage` for vyos-1x exits non-zero (e.g. upstream `Makefile:106` runs `PYTHONPATH=python/ python3 -m nose2 -v` whose `test_check_port_availability` probes 127.0.0.1:8080 and fails when the self-hosted Cobalt 100 runner has leftover lighttpd / accel-ppp-ng bound to it), build.py prints the "ignoring" line and continues, **NO `vyos-1x_*_arm64.deb` is produced**, `ci-pick-packages.sh` finds no vyos-1x .deb to stage, and live-build's `chroot_install` resolves vyos-1x from the **stock VyOS apt repo** — silently shipping an ISO with **completely unpatched `image_installer.py`**. Symptom on the board: `add system image` writes a boot dir under `/boot/<image>/` containing only `vmlinuz`+`initrd.img`+`*.squashfs`, with **NO `mono-gw.dtb`** (our patches `vyos-1x-006` and `vyos-1x-011` add the `Path(DIR_ISO_MOUNT).iterdir(): ... .dtb` copy loop to `add_image()`), so U-Boot's `bootcmd` falls through to SPI recovery and the install is unbootable until manually fixed at the U-Boot prompt. Two guards now exist: (1) `bin/ci-setup-vyos1x.sh` pre_build_hook ends with a `sed -i 's|^\tPYTHONPATH=python/ python3 -m nose2.*|\ttrue|' Makefile` that neuters the nose2 invocation while preserving the `python3 -m compileall` syntax check on the prior recipe line; (2) `bin/ci-build-packages.sh`'s post-build cache-populate loop hard-fails with `exit 1` and a `::error::` annotation if `../vyos-1x_*_arm64.deb` is empty. Verify in CI logs by searching for `Failed to build package vyos-1x` (must be **absent**) AND for `### Cached N vyos-1x .deb(s)` (must be **present**, N≥1). Verify on a fresh board install by `unsquashfs -l` of the shipped squashfs and `grep -E '\.dtb|mmcblk|fw_setenv|vyos\.env' usr/libexec/vyos/op_mode/image_installer.py` — zero hits means the substitution happened. Diagnosed from CI run 26142046765, 2026-05-20.
- **CI build-time caches (vyos-1x .deb + zstd squashfs):** Two optimizations together cut the warm-cache CI wall from ~15 min → ~7 min. (1) `bin/ci-build-packages.sh` caches the produced `vyos-1x_*_arm64.deb` under `${RUNNER_TOOL_CACHE:-/tmp}/vyos-1x-cache/` keyed by `vyos-1x_<UPSTREAM_SHA12>_<PATCH_HASH16>`, where `UPSTREAM_SHA` is parsed from the `commit_id = "..."` line of `vyos-build/scripts/package-build/vyos-1x/package.toml` and `PATCH_HASH` is `sha256(cat data/vyos-1x-*.patch) | cut -c1-16`. Cache HIT replays the `.deb` to `package-build/` (where `dpkg-buildpackage` outputs it — parent of the `vyos-1x/` source dir) and skips `./build.py` entirely (saves ~6 min). Cache MISS runs the build then populates the cache. **The patch-glob hash is intentionally sensitive to filename order** — renaming `data/vyos-1x-NNN-*.patch` files (or adding/deleting any of them) legitimately invalidates the cache because patch application order in upstream `build.py` is `sorted(patch_dir.glob('*'))` and a different order can produce a different `.deb`. 14-day mtime GC at the top of every run keeps the cache from growing unbounded. Self-hosted runner persistent disk on the Cobalt 100 ARM64 VM means the cache survives between runs. (2) `bin/ci-setup-vyos-build.sh` rewrites `vyos-build/data/defaults.toml`'s `squashfs_compression_type` from upstream's `"xz -Xbcj x86 -b 256k -always-use-fragments -no-recovery"` (single-threaded xz with an x86-only BCJ filter that is meaningless on ARM64) to `"zstd -b 1M -Xcompression-level 22"`. zstd `mksquashfs` is parallel-by-default on the 4-core Cobalt 100 → ~3–4× faster `mksquashfs` step; level 22 keeps squashfs within ~5–8% of xz size. The string passes verbatim through `lb config --chroot-squashfs-compression-type` via the Jinja2 template in `vyos-build/scripts/image-build/build-vyos-image` (there is no `auto/config` file upstream — the `lb config` invocation is rendered from that Jinja2 template, so we override at the `defaults.toml` source-of-truth level instead). **Do NOT delete the `if grep -q '^squashfs_compression_type'` guard** — it makes the sed idempotent across upstream `defaults.toml` resyncs.
- **Boot optimizations:** `acpid.service`, `acpid.socket`, `acpid.path` are masked in the ISO via `99-mask-services.chroot`. `kexec-load.service` and `kexec.service` are NOT masked — mainline 6.6 QBMan kexec fix enables VyOS managed-params self-healing on DPAA1. SysV init scripts (`/etc/init.d/kexec-load`, `/etc/init.d/kexec`) are removed to prevent `systemd-sysv-generator` from creating duplicate units that would bypass systemd. The old `ln -sf /dev/null` in `includes.chroot` approach was broken — live-build dereferences absolute symlinks to paths outside the chroot, producing empty files instead. ACPI masking saves ~2s. `CONFIG_DEBUG_PREEMPT` suppression saves ~20s. Installed system boot time: ~82s to login prompt.
- **`nopersistence` is REQUIRED for TFTP / USB live-boot, REMOVED for eMMC install (commit c689b96e):** The eMMC install path has `vyos-union=/boot/<image>` in U-Boot bootargs and uses live-boot's persistence branch (re-purposed by VyOS) to bind-mount `/config`, `/home`, `/opt`, etc. through `find_persistence_media()` → `get_custom_mounts()` → `activate_custom_mounts()`. With `nopersistence` set, that branch is short-circuited and the overlayfs upper layer falls back to tmpfs (RAM) — `commit save` writes `/config/config.boot` to RAM and reboot wipes it. Therefore `UBOOT_BOOTARGS_TAIL` in `board/scripts/vyos-postinstall` does NOT contain `nopersistence`. - **TFTP dev-loop bootargs** in `bin/dev-build.sh` (which preserves the same kernel cmdline as the legacy LXC 200 script) and the USB live-boot script `board/scripts/boot.cmd` DO retain `nopersistence` because those paths have no `vyos-union=` anchor and no eMMC ext4 partition to back the overlay — persistence is neither possible nor desired there. `nopersistence` is NOT a `MANAGED_PARAMS` value → no kexec double-boot risk on eMMC. The historical prober-stub hook `data/hooks/91-strip-persistence-prober.chroot` (which broke eMMC persistence by stubbing `find_persistence_media()` to `return 0`) was removed in c689b96e. If you re-introduce `nopersistence` in eMMC bootargs you will lose `/config/config.boot` across reboots — see "Boot Diagnostics" below for the residual `/init: line 1365` console message and how `data/hooks/92-livescripts-defensive-mount-list.chroot` handles it for the TFTP/USB paths.
- **Console verbosity uses `loglevel=4 systemd.show_status=true`, not `quiet`:** Stock VyOS amd64 GRUB uses `quiet` which sets kernel loglevel to 4 AND disables systemd's per-unit status lines. On our serial-console-only boards that produces ~30s of dead air between `Starting kernel ...` and the login banner (kernel quiet + systemd quiet + 16s spent in `live-config.service`). We pass `loglevel=4` (same kernel-side effect as `quiet`) plus explicit `systemd.show_status=true` (what `quiet` would otherwise turn off) so the operator sees `[ OK ] Started …` lines as boot progresses. If you ever need full kernel chatter for diagnostics, drop `loglevel=4` from `UBOOT_BOOTARGS_TAIL` in `board/scripts/vyos-postinstall`.
- **Chroot hooks under `data/hooks/*.chroot` do NOT auto-apply:** `data/hooks/` is a staging area only. `bin/ci-setup-vyos-build.sh` MUST explicitly `cp data/hooks/<name>.chroot "$HOOKS/<name>.chroot" && chmod +x` (where `$HOOKS=vyos-build/data/live-build-config/hooks/live`). live-build only runs scripts physically present in its own `hooks/live/` tree at `lb build` time. Existing wired hooks: `92-livescripts-defensive-mount-list`, `94-vbash-vyatta-env`, `95-vyos-hostname`, `96-enable-services`, `97-ask-modules` (unconditional — packages `ask.ko` when present, no longer FLAVOR-gated), `98-fancontrol`, `99-mask-services`. **When adding a new hook, update `data/hooks/` AND `bin/ci-setup-vyos-build.sh` in the SAME commit.** Symptom of forgetting: hook silently doesn't run; `gh run view <id> --log | grep <hook-name>` returns no echo lines from the hook body.
- **`configure` op-mode SIGABRTs on ttyS0 unless VYATTA_* env is primed at login:** `/etc/bash_completion.d/vyatta-cfg` (1200 lines, shipped by vyos-1x verbatim from upstream vyatta-cfg) early-returns at line 38 unless `_OFR_CONFIGURE=ok` is set in the environment. The op-mode `configure` (`/opt/vyatta/share/vyatta-op/templates/configure/node.def`) sets that flag and runs `newgrp vyattacfg`, which exec's a fresh vbash login shell. The post-newgrp shell's profile chain (`/etc/profile` → `~/.profile` → `~/.bashrc` → `/usr/share/bash-completion/bash_completion` → `/etc/bash_completion.d/vyatta-cfg`) is supposed to re-source vyatta-cfg with the flag still set, and line 126 `eval "$(my_cli_shell_api getSessionEnv $$)"` populates `VYATTA_TEMP_CONFIG_DIR` / `VYATTA_CONFIG_TMP` / `VYATTA_CHANGES_ONLY_DIR` followed by line 1157 `vyatta_cli_shell_api setupSession`. That chain is reliable over SSH-with-PTY but sporadically fails on the LS1046A ttyS0 serial console: `setupSession` aborts with `terminate called after throwing an instance of 'std::out_of_range' / basic_string::erase: __pos (which is 18446744073709551615) > this->size() (which is 0) / Failed to set up config session`. The crash is `UnionfsCstore::setupSession()` doing `work_string.erase(work_string.find_last_of("/"))` with `work_root` empty (npos `erase` throws). `work_root` is derived from `VYATTA_TEMP_CONFIG_DIR`, which gets unset because the getSessionEnv eval produced an empty string — likely a startup race specific to the serial-console path. **Fix:** `data/hooks/94-vbash-vyatta-env.chroot` drops `/etc/profile.d/zz-vyatta-cfg-env.sh` which force-sources vyatta-cfg with `_OFR_CONFIGURE=ok` on every interactive vbash login, populating VYATTA_* directly in the user's login shell. `configure`'s subsequent `newgrp` then inherits the primed env via `$PPID` lookup in script-template's `vyatta_configure()`. Do **not** patch the upstream vyatta-cfg file directly — additive `/etc/profile.d/` drop-in avoids a 1200-line rebase burden on every VyOS upstream pull. **LOGIN RACE (fixed 2026-06-11):** the drop-in MUST gate on `/tmp/vyos-config-status` before sourcing vyatta-cfg. Historically the board's serial-getty had its upstream `ExecStartPre=vyos-config SERIAL` readiness gate fully cleared (`zz-ls1046a-nodevbind.conf` in hook 96), so the login prompt showed at ~T+28s while the config tmpfs mounts ~T+45s and the boot commit finishes ~T+74s. A login in that window made the drop-in source vyatta-cfg → `setupSession` fails → vyatta-cfg line ~1158 runs `builtin exit 1`, which in a SOURCED profile **kills the login shell** ("Failed to set up config session", straight back to `login:`, hostname stuck at `localhost`, soft-lock until reboot; `|| true` on the source line cannot catch a builtin exit). The drop-in now re-creates upstream's gate inside the login shell: bounded wait (120 s, visible "Waiting for VyOS boot configuration to complete..." message) for `/tmp/vyos-config-status` — the same marker the stock getty gate uses — plus a `-w /opt/vyatta/config/tmp` guard; on timeout it skips priming instead of dying. Verified on hardware 2026-06-11 (serial login 5 s after prompt: wait message → `done.` → successful `vyos@vyos` login, no soft-lock). **GETTY GATE RESTORED IN BOUNDED FORM (2026-06-11):** the prompt-too-early side was then fixed in hook 96 — `zz-ls1046a-nodevbind.conf` re-adds an `ExecStartPre` that waits (bounded, 180 s, `TimeoutStartSec=240`) for the same `/tmp/vyos-config-status` marker, so the serial prompt now appears only AFTER the boot commit (~T+74s) like stock VyOS, with no late vyos-router log lines printing over it. Upstream's wait is UNBOUNDED — we bound it so a hard vyos-router crash (marker never written) still yields a prompt at ~T+200s on a serial-console-only board. Do NOT gate the getty on `multi-user.target` instead: `vyos-router.service` is `Type=simple`+`RemainAfterExit=yes` and reports active ~80 ms in while the boot commit runs asynchronously, so multi-user.target is reached at ~T+20s — long before the config is loaded (and `After=multi-user.target` on a getty creates an ordering cycle via `Before=getty.target`). The marker file is the ONLY true completion signal. The profile-side 120 s wait is kept as a safety net (vyos-router crash, SSH logins that beat the marker, emergency console sessions). HW-verified 2026-06-11 (ISO 2026.06.11-1500-rolling, serial boot capture): getty unit started T+63s with gate waiting silently, multi-user.target reached T+88s (43 s before config load — proving target-gating would fire too early), prompt appeared T+131s exactly at boot-commit completion with zero log lines after it, auto-login 2 s after prompt succeeded cleanly (single banner, hostname correct, no soft-lock).
- **`Failed to set up config session` can ALSO be a permissions bug, distinct from the env-priming case above:** if `cli-shell-api getSessionEnv $$` SUCCEEDS (emits a correct `VYATTA_TEMP_CONFIG_DIR`) but `cli-shell-api setupSession` returns rc=1 and never creates `/opt/vyatta/config/tmp/new_config_<pid>`, the cause is `/opt/vyatta/config/tmp` being `root:root` mode 775 instead of `root:vyattacfg` setgid. Upstream `/usr/libexec/vyos/init/vyos-router` (~line 588) mounts the `/opt/vyatta/config` tmpfs `mode=775` (NO setgid bit) and `chgrp vyattacfg` ONLY the top-level dir, so the `tmp/` and `active/` subdirs created later inherit the creating process's gid (root) instead of vyattacfg. The `vyos` user is in group `vyattacfg` but NOT root, so its per-session `mkdir /opt/vyatta/config/tmp/new_config_<pid>` is denied → setupSession cannot create the session dir → the unionfs-fuse mount aborts with `Failed to open .../changes_only_<pid>/: No such file or directory` → rc=1 → "Failed to set up config session". On amd64 this usually works because the subdir creators happen to run as vyattacfg; on this board they ran as root, exposing the missing setgid bit. **Fix:** `data/hooks/96-enable-services.chroot` installs a dedicated `ls1046a-config-perms.service` oneshot (enabled into `multi-user.target.wants`) that forces `/opt/vyatta/config{,/tmp,/active}` to group `vyattacfg` WITH the setgid bit (2775) on every boot, so every future per-pid session dir inherits vyattacfg. **The unit MUST be `After=vyos-router.service`, NOT an `ExecStartPost=` drop-in on vyos-router itself** — the original `zz-ls1046a-config-perms.conf` ExecStartPost approach was proven RACY and INEFFECTIVE on 2026-05-30 (the board at 192.168.1.190, ISO 2026.05.30-0515-rolling): the journal showed vyos-router's ExecStartPost firing at `05:35:39` with `chgrp/chmod: cannot access '/opt/vyatta/config/tmp': No such file or directory` while the tmpfs mount + `tmp/`+`active/` creation happened **16 s later** at `05:35:55` (`Mounting VyOS Config...done.`). The `/opt/vyatta/config` tmpfs is mounted and populated by an ASYNCHRONOUS stage inside vyos-router that completes AFTER the unit's main `ExecStart` returns, so `ExecStartPost` runs too early, the `-`-swallowed chgrp/chmod hit non-existent paths, and `tmp/`+`active/` stayed `root:root` 775. A separate `After=vyos-router.service` oneshot alone is **STILL not enough** (proven 2026-06-11, board journal): vyos-router reports `active` ~80 ms after start while the tmpfs mounts ~17 s later, AND `/opt/vyatta/config/tmp` already exists in the squashfs UNDERLAY, so even a dir-existence poll passes instantly and the chmods land on the underlay just before the tmpfs mounts over them. The unit's first `ExecStart` therefore polls **`mountpoint -q /opt/vyatta/config`** (plus `-d tmp/`, up to 240 s, `TimeoutStartSec=300`) before applying chgrp/chmod — that is the only reliable gate. The hook also `rm -f`s the superseded `vyos-router.service.d/zz-ls1046a-config-perms.conf` so in-place upgrades drop the broken drop-in. Live remediation (no reboot): `sudo chgrp vyattacfg /opt/vyatta/config/tmp /opt/vyatta/config/active && sudo chmod 2775 /opt/vyatta/config/tmp /opt/vyatta/config/active`. To distinguish from the env-priming `std::out_of_range` crash above: check that `getSessionEnv` succeeds (non-empty `VYATTA_TEMP_CONFIG_DIR`) and that `ls -ld /opt/vyatta/config/tmp` shows `root root` (not `root vyattacfg`) — that is the permissions case. Diagnosed 2026-05-29 on the board (192.168.1.190) (ISO 2026.05.29-2226-rolling, kernel 6.18.33-vyos); ExecStartPost-race correction 2026-05-30 (same board, ISO 2026.05.30-0515-rolling).
- **`CONFIG_NET_SCH_FQ=y` invariant:** The base VyOS arm64 vyos_defconfig has `CONFIG_NET_SCH_FQ=m`. With `=m`, `/etc/sysctl.d/*` writes `net.core.default_qdisc=fq` during the systemd very-early sysctl pass (well before `systemd-modules-load.service` loads `sch_fq.ko`); the kernel proc handler validates against currently-registered qdiscs and returns `-ENOENT`, leaving the qdisc silently at `pfifo_fast`. The in-tree fragment `kernel/common/vyos-base/10-networking.config` overrides to `=y` so `sch_fq` is registered before sysctl runs. **Do not add a runtime workaround** (custom systemd ordering, `modprobe` in a hook) — this must stay a build-time kernel-config fix.

## Boot Diagnostics (Ignore These)

- **`smp_processor_id() in preemptible code: python3`** — Suppressed via `# CONFIG_DEBUG_PREEMPT is not set` in defconfig. If seen on older builds: cosmetic only, PREEMPT_DYNAMIC on Cortex-A72.
- **`could not generate DUID ... failed!`** — Expected on live boot without persistence (no stable machine-id)
- **`WARNING failed to get smmu node: FDT_ERR_NOTFOUND`** — DTB lacks SMMU/IOMMU nodes. Harmless.
- **`PCIe: no link` / `disabled`** — No PCIe devices on the board. Normal.
- **`bridge: filtering via arp/ip/ip6tables is no longer available`** — `br_netfilter` not loaded. VyOS loads it when needed.
- **`nfct v1.4.7: netlink error: Invalid argument`** — Conntrack helper setup during TFTP dev boot. Cosmetic, first boot only — kexec replaces with eMMC kernel.
- **`binfmt_misc.mount` FAILED** — Expected on ARM64 target hardware. No binfmt emulation needed.
- **`mount: /live/persistence/ failed: No such device`** — On the eMMC install path (post c689b96e) this is no longer suppressed; `find_persistence_media()` runs as upstream intends and discovers `mmcblk0p3` for the `vyos-union=` overlay. The per-candidate `mount: ... failed` lines for the OTHER candidate block devices (`mmcblk0boot0`, `mmcblk0boot1`, `mmcblk0p1`, etc.) are normal probe output — only `mmcblk0p3` is expected to succeed. On TFTP / USB live-boot paths `nopersistence` short-circuits the prober entirely so this message does not appear there.
- **`/init: line 1365: can't open /tmp/custom_mounts.list: no such file`** — On the eMMC install path this is no longer expected: `get_custom_mounts()` populates the list normally now that `find_persistence_media()` runs. On TFTP / USB live-boot paths (`nopersistence` set), `data/hooks/92-livescripts-defensive-mount-list.chroot` patches `activate_custom_mounts()` to pre-create an empty list file on entry, so the `done < $custom_mounts` redirection becomes a no-op instead of an init-time error. If you see this message on a current eMMC build, either the U-Boot env still carries `nopersistence` (re-run `vyos-postinstall`), or the live-boot/initrd was built without the 92 hook (`gh run view <id> --log | grep 92-livescripts-defensive-mount-list` should show the patched-marker echo).
- **`Error -ENOENT writing to proc file to set sysctl parameter 'net.core.default_qdisc=fq'`** — Indicates `CONFIG_NET_SCH_FQ` ended up `=m` in the final kernel config. Verify `kernel/common/vyos-base/10-networking.config` contains `CONFIG_NET_SCH_FQ=y` after `kernel/common/scripts/stage-kernel.sh --flavor <flavor>`. Also verify `bin/ci-setup-kernel.sh` is `sed`-patching `vyos-build/scripts/package-build/linux-kernel/config/13-net-sched.config` to `=y` (vyos-build's `merge_config.sh` runs that fragment AFTER ours and otherwise overrides `=y` back to `=m`).
- **`sfp-xfi0: deferred probe pending`** — SFP cages wait for PHY driver. Resolves after full boot.
- **`can't get pinctrl, bus recovery not supported`** — I2C pinctrl not in DTB. Harmless.
